ChemVault Docs is not only a public documentation site. It is the operating surface for a chemistry knowledge product: users should be able to search, decide, prepare, publish and maintain from a consistent system.

## Product thesis

ChemVault Docs should behave like a **ChemVault Knowledge OS**:

- It captures chemistry knowledge in stable workspaces.
- It structures pages by audience, route and release state.
- It turns notes into decisions through maps, workflows and troubleshooting pages.
- It publishes resources only when context, metadata and maintenance rules exist.

## Operating loop

1. **Capture** a note, workflow, reagent fact or product decision.
2. **Structure** it into a workspace with a title, status, route purpose and next action.
3. **Connect** it to adjacent pages so users can move from question to answer.
4. **Publish** it only when links, metadata and build checks are clean.
5. **Record** important changes in the changelog.

## Commercial quality bar

- The first screen should explain the product, not just list folders.
- Every major section should have a job: learn, decide, execute, publish or maintain.
- Public claims should be conservative and backed by visible routes.
- Future SaaS features should appear first as clear docs surfaces before becoming platform features.

## Current gaps to close

- More reagent passports for common transformation decisions.
- More resource candidates with complete metadata.
- More cross-links from CHEM10712 notes into concept and lab pages.
- A stronger searchable taxonomy as the library grows.
