ChemVault Extract 用于把科学论文、实验报告、仪器导出、表格和文本文件转成 evidence-backed research database。它的核心原则是：AI 可以提取候选记录，但正式数据需要人工审核。

## 入口

- Web App：[https://app.chemvault.science](https://app.chemvault.science)
- API Base URL：[https://api.chemvault.science](https://api.chemvault.science)

## 适合处理的文件

- PDF
- DOCX
- CSV
- XLSX
- TXT
- Markdown

## 不适合直接处理的内容

- 没有文本层的扫描图片 PDF，除非已经经过 OCR。
- 完全没有来源说明的摘录片段。
- 只包含图片、压缩包或二进制仪器原始文件的文档。
- 需要人工解释实验安全、医学结论或法律结论的内容。
- 过大的混合文件，建议先拆分成论文、支持信息、表格或附录。

## 核心流程

1. 登录账号。
2. 创建 workspace 或 project。
3. 上传文档。
4. 解析文档并生成 chunks。
5. 估算 AI extraction 成本。
6. 运行 AI extraction。
7. 在 Review Workflow 中审核、修正、批准或拒绝记录。
8. 搜索 approved records。
9. 导出 CSV、JSON 或 XLSX。

## Dashboard

Dashboard 是 Extract 的操作首页。常见入口包括：

- Upload document：上传单个文档。
- Run batch extraction：批量运行提取。
- Review records：审核候选记录。
- Search database：检索已解析文档和已审核记录。
- Export data：导出数据集。
- Usage：查看使用量、计划限制和消耗状态。

## Upload Documents

上传时需要选择项目，并提交文件。上传后系统会创建 Document row 和 parse job。

注意事项：

- 不支持的文件扩展名会在存储前被拒绝。
- 文档解析完成后才适合进行 AI extraction。
- DOCX 支持能力可能随版本阶段变化，遇到 not implemented 提示时应换用 PDF、TXT、CSV 或 XLSX。

## AI Extraction

AI Extraction 会把选中的 document chunks 送入提取流程，而不是默认发送完整原始 PDF。系统优先处理 Experimental、Methods、Results、Supporting Information、tables 等科学信息密度高的片段。

使用建议：

- 提取前先查看 cost estimate。
- 超出 plan limit 或 monthly usage limit 时，任务不会继续排队。
- 不要把 AI 输出直接当作正式记录。

## 可提取的信息类型

| 类型 | 示例字段 | 审核重点 |
| --- | --- | --- |
| Compound | name、formula、SMILES、identifier、role | 名称是否来自原文，结构标识是否可靠。 |
| Reaction | substrate、reagent、solvent、temperature、time、yield | 单位、条件和产率是否对应同一实验。 |
| Method | protocol、instrument、parameter、sample | 方法步骤是否完整，参数是否缺失。 |
| Spectroscopy | NMR、IR、MS、peak、assignment | 峰值、单位和结构归属是否一致。 |
| Material | composition、property、measurement | 属性和测试条件是否同时保留。 |
| Table Row | row label、value、unit、source table | 表格行列是否错位，单位是否混淆。 |

## 字段解释

| 字段 | 含义 |
| --- | --- |
| source document | 候选记录来自哪个文档。 |
| page number | 原文页码或解析页码。 |
| section | Methods、Results、SI、Table 等来源区域。 |
| evidence quote | 支持该字段或记录的原文引用。 |
| raw value | 原文中的原始值。 |
| normalized value | 经过单位、名称或格式整理后的值。 |
| confidence | 系统估计的可信度，不等于人工确认。 |
| validation warning | 系统发现的字段异常、缺失或冲突提示。 |

## Review Workflow

Review Workflow 是 Extract 的质量控制中心。

常见状态：

- `pending`：等待审核。
- `needs_review`：需要人工判断或修正。
- `approved`：已批准，可进入正式数据集或导出。
- `rejected`：已拒绝，不应进入默认导出。

审核时应检查：

- source document
- page number
- section
- evidence quote
- confidence
- validation warning
- raw value 和 normalized value

## Search

Search 页面用于查询 documents、chunks 和 reviewed scientific records。

建议：

- 默认使用 approved records 作为正式检索对象。
- 需要排查提取结果时，再查看 pending 或 needs_review 的记录。
- 用 source、document、record type 或关键词缩小结果。

## Export

Export 用于把审核后的数据交给下游分析。

支持格式：

- CSV
- JSON
- XLSX

导出建议：

- 默认只导出 approved records。
- 导出前确认数据集版本、项目、记录类型和字段定义。
- 需要证据链时，保留 source quote、page、section 和 confidence 字段。

## 数据质量检查

导出前建议抽查：

- 每个关键字段都有 evidence quote。
- 单位已经统一，例如 mg、g、mL、M、C、h。
- 同一化合物没有多个互相冲突的名称。
- yield、conversion、selectivity 等数值没有混用。
- 表格数据没有发生列错位。
- rejected 记录没有进入默认导出。
- approved 记录能追溯到原始文档。

## Developer API

开发者可以用 API 上传文件、查询项目、运行提取、读取记录和创建导出。

常见 scope：

- `projects:read`
- `projects:write`
- `documents:read`
- `documents:write`
- `extractions:read`
- `extractions:write`
- `records:read`
- `exports:read`
- `exports:write`

API Key 应按最小权限创建，并定期撤销不再使用的 key。

## Webhooks

Webhook 用于接收 Extract 的异步事件。

常见事件：

- `document.uploaded`
- `document.parsed`
- `document.parse_failed`
- `extraction.started`
- `extraction.completed`
- `extraction.failed`
- `normalization.completed`
- `review.item_created`
- `review.item_approved`
- `review.item_rejected`
- `export.completed`
- `batch.completed`

安全要求：

- 每个 delivery 使用 HMAC SHA256 签名。
- 接收端应使用 raw request body 验签。
- 不要在 webhook payload 中传输 OpenAI API Key、密码或完整原始文件。

## 常见排障

| 现象 | 处理方式 |
| --- | --- |
| 上传成功但不能提取 | 检查 parse job 是否完成。 |
| cost estimate 过高 | 缩小文档范围，优先处理 Methods、Results 或表格。 |
| 审核队列过长 | 先按低 confidence、warning 或目标 record type 筛选。 |
| 导出结果为空 | 确认是否有 approved records。 |
| API 返回 401 | 检查 API Key 是否存在、是否被撤销。 |
| API 返回 403 | 检查 scope、项目权限和服务访问。 |
